從「Reasoning 思維」到「Agentic 思維」

Junyang Lin (@JustinLin610)

過去兩年徹底改變了我們評估模型、以及對模型的期待。OpenAI 的 o1 證明「思考」可以是模型的第一級能力，可以被訓練、被公開給使用者；DeepSeek‑R1 則證明這種 reasoning 型後訓練能在原始實驗室之外被重現並擴展。OpenAI 把 o1 描述成以 RL 訓練、能「先想再答」的模型；DeepSeek 把 R1 定位為可與 o1 競爭的開放 reasoning 模型。

那一段很重要，但 2025 上半年仍多數停留在 reasoning 思維：如何讓模型花更多推理算力、如何給更強的 reward、如何控制或曝光這段額外推理。下一步是什麼？我認為答案是 agentic 思維：為了「行動」而思考，與環境互動，並基於世界回饋持續修正計畫。

[!note] 註記 Reasoning 重點在「想得更深」，Agentic 重點在「想了要能持續有效行動」。

1. o1 與 R1 的崛起教會我們什麼

第一波 reasoning 模型告訴我們：若要在語言模型中擴展 RL，就需要可驗證、穩定、可擴展的回饋訊號。數學、程式、邏輯等可驗證領域變成核心，因為這些場景的 reward 強度遠高於泛用偏好監督。RL 能優化「正確性」而非「看起來合理」。基礎設施（infra）變得關鍵。

一旦模型被訓練成能走長推理軌跡，RL 就不再是 SFT 的小補丁，而是系統工程問題：需要大規模 rollouts、吞吐量高的驗證、穩定的 policy 更新、有效率的採樣。Reasoning 模型的出現，不只是一個建模故事，更是一個 infra 故事。o1 與 R1 清楚指出：後訓練 scaling 才是下一個重點。

第一個大轉折：從擴展 pretraining → 擴展 post‑training 的 reasoning。

2. 真正的問題從來不是只把「Thinking」和「Instruct」合併

2025 初，Qwen 團隊想像的理想系統是：統一 thinking 與 instruct，提供可調 reasoning effort（低 / 中 / 高），最好還能自動推斷該用多少推理。概念方向對，但難點在 資料。

人們談「合併」時，往往只想到 model 端：同一 checkpoint 能否支援兩種模式、是否能在 chat template 切換、serving 是否能提供開關。但更深層的問題是：兩種模式的資料分佈與行為目標大不同。

  • Instruct 模型獎勵的是：直接、簡潔、格式合規、低延遲，適合大量重複任務（改寫、標註、模板化支援、結構化抽取、運維 QA）。
  • Thinking 模型獎勵的是：在困難問題上花更多 token、維持中間結構、探索替代路徑、提高最終正確率。

這兩種行為互相拉扯。若合併資料沒有精心設計，結果通常是兩邊都變得中庸：thinking 變得冗長雜訊化，instruct 變得不夠俐落、成本高。

實務上，分離仍很有吸引力。Qwen3 之後的 2507 線推出獨立的 Instruct 與 Thinking（30B / 235B）。在商業部署中，許多客戶仍偏好高吞吐、低成本、可高度可控的 instruct 行為。分開反而更能聚焦各自的資料與訓練問題。

但也有實驗室走相反路線：

  • Anthropic：主張整合式模型哲學，Claude 3.7 Sonnet 為 hybrid reasoning，可選一般或 extended thinking，API 可設 thinking budget。
  • GLM‑4.5：公開定位為 hybrid reasoning（含 reasoning / coding / agent 能力）。
  • DeepSeek V3.1：提出「Think & Non‑Think」混合推理。

關鍵問題是：合併是否自然。若只是把兩種人格硬塞進同一 checkpoint，體驗仍不自然。真正成功的合併必須是「推理強度光譜」平滑連續，模型能表現多個 effort 等級，最好還能自適應選擇。這是 GPT‑style effort control 的方向：控制計算策略，而不是二元開關。

3. Anthropic 的方向為什麼重要

Anthropic 對 Claude 3.7 / Claude 4 的公開敘事相當克制：整合 reasoning、可控 thinking budget、真實任務、coding 品質、以及在 extended thinking 中可用工具。Claude 4 將 thinking 與 tool use 交錯，並強調 coding、長任務、agent workflow。

更長的 reasoning trace 不代表更聰明。過度冗長的可見推理，反而是分配失衡的訊號。Anthropic 傳遞的觀點是：思考必須由目標工作負載決定。

  • 如果目標是 coding：thinking 應該幫助 codebase navigation、規劃、拆解、錯誤修復、工具協調。
  • 如果目標是 agent workflow：thinking 應該提升長時程執行品質，而非產生好看的中間長文。

這指向更大的轉變：我們正在從「訓練模型」走向「訓練代理」。

Qwen3 blog 也明確寫過：「我們正從訓練模型轉向訓練 agents。」RL 的下一步會連結到環境回饋與長時程推理。Agent 是能制定計畫、決定何時行動、使用工具、感知環境回饋、修正策略並長時間持續的系統。它的本質是 closed‑loop 與世界互動。

4. 什麼是「Agentic 思維」

Agentic 思維是另一個優化目標。Reasoning 思維常以「內部推理品質」衡量：能否解定理、寫證明、產出正確 code、通過 benchmark。Agentic 思維則關注：在與環境互動時，能否持續推進任務。

核心問題從「模型能不能想夠久？」變成： 「模型能不能用能支撐有效行動的方式去想？」

Agentic 思維必須處理 reasoning 模型多半能避開的事情：

  • 何時停止思考並採取行動
  • 選擇哪個工具、以什麼順序調用
  • 吸收噪音或不完整的環境觀測
  • 在失敗後修正計畫
  • 多輪、多工具調用下保持一致性

Agentic 思維 = 一個能透過行動推理的模型。

5. Agentic RL 基礎設施更難

當目標從解 benchmark 轉到互動任務，RL 堆疊就變了。傳統 reasoning RL 的 infra 不夠用。Agentic RL 的 policy 嵌在更大的 harness 中：工具伺服器、瀏覽器、終端、搜尋、模擬器、執行沙盒、API 層、記憶系統、編排框架。環境不再只是 verifier，而是訓練系統的一部分。

這帶來新的系統需求：訓練與推理要更清楚地解耦。否則 rollout 吞吐量會崩潰。想像一個 coding agent 必須執行程式並等待測試回饋：推理端被卡住，訓練端等不到完整軌跡，整體 GPU 利用率遠低於傳統 reasoning RL。工具延遲、部分可觀測、狀態式環境會放大這些效率損失。

環境本身也成為研究成果。 SFT 時代我們迷戀資料多樣性；agent 時代我們應該迷戀環境品質：穩定性、真實性、覆蓋度、難度、狀態多樣性、回饋豐富性、抗 exploit 能力、rollout 可擴展性。建環境開始成為一個真正的 startup 類別，而不是 side project。

6. 下一個前沿：更可用的思考

我預期 agentic 思維會成為主流，最終取代很多傳統 reasoning 思維的「長篇獨白式推理」——那種在缺乏互動時用大量文字補償的模式。即使是很難的數學或 coding 任務，一個真正先進的系統也應該能搜尋、模擬、執行、檢查、驗證、修正。

最難的挑戰是 reward hacking。一旦模型能用工具，reward hacking 的風險就會大增：

  • 有搜尋能力的模型可能直接查答案
  • coding agent 可能利用 repo 的未來資訊、濫用 log、或找到「捷徑」讓任務失效
  • 有漏洞的環境會讓模型看起來超人，其實是在作弊

這讓 agent 時代比 reasoning 時代更脆弱。工具越強大，攻擊面越大。未來的研究瓶頸將來自：環境設計、評估器魯棒性、反作弊協議、以及 policy 與世界的介面設計。

但方向很清楚：具工具能力的思考比孤立思考更有用，且更能提升真實生產力。

Agentic 思維也意味著 harness engineering：核心智慧越來越多來自「多代理如何組織」——一個 orchestrator 負責規劃與路由、專家代理處理領域任務、子代理負責更窄的執行與 context 隔離。未來是 訓練模型 → 訓練代理 → 訓練系統。

結論

第一階段的 reasoning 浪潮證明了：在可靠回饋訊號與足夠 infra 下，RL 能顯著提升模型認知能力。

更深的轉變則是：從 reasoning 思維 → agentic 思維。訓練的核心對象不再是單一模型，而是「模型 + 環境」的系統，也就是 agent 與其 harness。這改變了研究中最重要的工件：除了模型架構與訓練資料，還包含環境設計、rollout infra、評估器魯棒性、多代理協調介面。

它也改變了「好思考」的定義：最有價值的思考，是能在真實世界限制下維持行動的思考，而不是最長或最華麗的推理痕跡。

競爭優勢也將轉移：在 reasoning 時代，優勢來自更好的 RL 演算法、更強的回饋訊號、更可擴展的訓練 pipeline；在 agentic 時代，優勢會來自更好的環境、更緊密的 train‑serve 整合、更強的 harness engineering、以及更快速閉環的決策回饋。
